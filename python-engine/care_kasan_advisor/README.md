# 介護報酬加算 提案システム - ロジック設計

介護施設の基本情報(業務形態・人員配置・利用者構成・現有加算・現有システム)を入力すると、
獲得可能な介護報酬加算を「費用対効果」と「獲得難易度」の2軸で評価し、
① 費用対効果ランキング / ② コスト見積もり / ③ ロードマップ / ④ 獲得容易性ランキング
を出力するアルゴリズムのプロトタイプ実装。

## ファイル構成

| ファイル | 役割 |
|---|---|
| [models.py](models.py) | 入力データモデル(施設プロファイル)と加算マスタの型定義 |
| [kasan_master.py](kasan_master.py) | 加算マスタ(知識ベース)のサンプルデータ 13件 |
| [advisor.py](advisor.py) | コアアルゴリズム(フィルタリング→判定→コスト/収益試算→スコアリング→ランキング→ロードマップ生成) |
| [demo.py](demo.py) | 通所介護・特養の2プロファイルでの実行例(赤字ケースを含む) |

実行方法: `python demo.py`

---

## 1. 全体の処理フロー

```
入力(FacilityProfile)
   │
   ▼
①フィルタリング  … 業務形態(施設/通所/訪問/その他)に存在しない加算を除外
   │
   ▼
②ステータス判定  … 各加算の要件(Requirement)を現状データと突合し4区分に分類
   │                 取得済み / 即時申請可能 / あと一歩 / 要体制整備
   ▼
③コスト試算      … 初期コスト・年間ランニングコストを積み上げ計算
   │
   ▼
④収益試算        … 加算の算定単位(日額/月額/事業所単位/回数/報酬総額に対する%)に応じて
   │                 月間・年間の加算収益を試算
   ▼
⑤難易度スコアリング … 人員変更要否・システム変更要否・書類負荷・リードタイム・未充足要件割合を重み付け加算
   │
   ▼
⑥ランキング生成    … ①費用対効果Top5 / ④獲得容易性Top5 / 赤字見込みの参考リスト
   │
   ▼
⑦ロードマップ生成  … 未充足要件→標準手順テンプレート→現有システムとの相性診断
   │
   ▼
レポート出力(Report)
```

## 2. 入力データモデル ([models.py](models.py))

仕様書のA〜Eにそのまま対応させている。

- `ServiceCategory` … A. 業務形態(施設型/通所型/訪問型/その他)
- `StaffInfo` … B. 人員配置(介護職員常勤換算・看護師・管理栄養士・PT/OT/ST・ケアマネ等)
- `UserComposition` … C. 利用者ステータス(要支援/要介護人数、認知症自立度、医療依存度)
- `CurrentStatus` … D. 取得済み加算・要件をほぼ満たしている加算
- `FacilityProfile.current_system` … E. 利用中の業務システム(カイポケ/ワイズマン/ほのぼの/ケアカルテ等)

## 3. 加算マスタ ([kasan_master.py](kasan_master.py))

各加算は `KasanDefinition` として以下を保持する、いわば「加算ごとのルールブック」。

- `service_categories` / `service_types` … どの業務形態・サービス種別に適用可能か(①のフィルタリングに使用)
- `billing_basis` + `unit_points` … 収益計算方法(日額×対象者数、月額、事業所単位、回数、報酬総額に対する%)
- `target_user_selector` … 加算の対象となる利用者数を利用者構成から算出する関数
  (例: 中重度者ケア体制加算は全利用者、特別管理加算は医療依存度の高い利用者のみ)
- `requirements` … 算定要件を「入力データに対する述語(True/False関数)」として実装
- `initial_cost_items` / `running_cost_items` … 初期コスト・ランニングコストを
  「現状の人員配置に応じて発生するかどうか」を条件付きで計算する関数のリスト
  (例: 管理栄養士が既に1名いれば追加人件費は0円、いなければ新規雇用コストを計上)
- `difficulty` … 難易度スコアリングに使う4要素(人員変更要否・システム変更要否・書類負荷1-5・リードタイム月数)
- `system_compatibility` … 現有システム(カイポケ等)ごとの対応可否コメント
- `roadmap_template` … 要件クリアの標準手順

> **重要**: 本マスタの単位数・要件は令和6年度(2024年度)介護報酬改定を基にした**代表的なサンプル**です。
> 地域区分による単価差や、その後の軽微な告示改正・Q&Aまでは反映していません。
> 実運用では厚生労働省の告示・通知および国保連資料と必ず突合し、`kasan_master.py` を
> **改定ごとに更新するデータとして管理**してください(バージョン管理を推奨)。

## 4. 各出力ロジックの詳細 ([advisor.py](advisor.py))

### ステータス判定 (`classify_status`)

| ステータス | 判定条件 |
|---|---|
| 取得済み | `current_status.acquired` に含まれる |
| 即時申請可能 | 全要件を充足(未充足要件0件) |
| あと一歩 | `near_acquisition` に含まれる、または未充足要件割合が34%以下 |
| 要体制整備 | 上記以外(未充足要件が多い) |

### ① 費用対効果ランキング

```
年間収益(annual_revenue) = 月間収益 × 12
初年度純損益(net_annual_benefit) = 年間収益 - 年間ランニングコスト - 初期コスト(初年度に全額計上)
```

未取得の加算のうち `net_annual_benefit > 0` のものを純損益降順で並べ Top 3〜5 を提示。
初期コストを初年度に全額計上する保守的なモデルにしており、「初年度は赤字だが2年目以降は黒字」
というケースも `annual_running_cost` と `initial_cost` を分けて表示することで判別できる。

### ② コスト見積もり

初期コスト・ランニングコストは「現状の人員配置」を参照して条件分岐で計算する。
例えば管理栄養士が既に配置されていれば採用コスト・人件費は計上されない(0円)。
これにより「専門職を新規に雇う必要があるか」を自動判定し、過大/過小見積もりを防いでいる。

### ③ ロードマップ

1. 未充足の算定要件(`Requirement`)を「要件充足」タスクとして先頭に列挙
2. 加算固有の標準手順(`roadmap_template`: アセスメント実施頻度、計画書作成、体制届出など)を続ける
3. 現有システム(`current_system`)ごとの対応可否コメントを付与し、
   「システムでそのまま対応可能」か「運用の変更・手動対応が必要」かを明示する

### ④ 獲得容易性ランキング

難易度スコア(低いほど容易)を次の重み付けで算出:

```
score = 30×(人員変更が必要か)
      + 20×(システム変更が必要か)
      + 6 ×書類作成負荷(1-5)
      + 4 ×リードタイム(月数)
      + 20×未充足要件割合
```

未取得の加算をスコア昇順で並べ Top 3〜5 を提示。「追加投資や人員変更がほぼ不要」な加算
(例: 口腔・栄養スクリーニング加算、特定事業所加算(II)のような書類・運用面の加算)が
自然と上位に来る設計。

## 5. 実行例で確認できること

`python demo.py` を実行すると、以下がログに出力される(要求仕様3.への対応確認):

- 通所介護(PT配置済・管理栄養士未配置)の例では、個別機能訓練加算(I)が
  追加投資ゼロで年間約739万円の収益と算出され①の1位になる一方、
  栄養アセスメント加算は外部管理栄養士との連携コストにより赤字(-110,000円)と算出される。
- 特養(小規模・管理栄養士未配置)の例では、栄養マネジメント強化加算について
  「年間加算収益 約108万円 に対し、管理栄養士 新規雇用の年間人件費 420万円 + 初期採用費 30万円が
  かかるため、初年度収支は約342万円の赤字」という、要求仕様の例そのままのアウトプットが出る。

## 6. 拡張方法

新しい加算を追加する場合は `kasan_master.py` に `KasanDefinition` を1件追加し、
`ALL_KASAN` リストへ加えるだけで、フィルタリング・判定・ランキング・ロードマップ生成の
ロジックは自動的に対応する(ロジック側の変更は不要)。

今後の拡張候補:
- 地域区分による `point_value_yen` の自動判定(都道府県・市区町村マスタとの連携)
- 加算間の依存関係(例: 科学的介護推進体制加算のLIFE提出体制が自立支援促進加算の前提)を
  グラフ構造で表現し、ロードマップを「前提加算→本命加算」の順に並べる
- 損益分岐点(何ヶ月で初期コストを回収できるか)の算出
- 実際の請求データ(カイポケ等からのCSVエクスポート)を読み込み `FacilityProfile` を自動生成する連携機能
