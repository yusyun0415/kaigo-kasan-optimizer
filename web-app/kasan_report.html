設計方針・全体アーキテクチャ

加算の単位数・算定要件は3年ごとの報酬改定(直近は令和6年度改定)と毎年の処遇改善加算見直しで変動する。したがって本設計では「加算マスタ」をロジックから完全分離し、マスタ(JSON/DB)だけ更新すればコード変更なしに追随できる構造とする。

処理の流れは、業務形態によるフィルタリングを起点に、現状の人員・利用者構成と各加算要件の差分(ギャップ)を測定し、そのギャップから収益・コスト・難易度を導出する一本のパイプラインとして設計する。

STEP 0形態フィルタ業務形態に存在しない加算を除外
STEP 1ギャップ分析人員・書類・システム要件との差分測定
STEP 2収益試算年間請求増加額を算出
STEP 3コスト試算初期・ランニングコストを算出
STEP 4費用対効果年間純利益で順位付け
STEP 5難易度評価ギャップの重み付き合成
STEP 6ロードマップ要件クリアの手順を生成

1

データモデル

1.1 入力スキーマ(FacilityInput)

ユーザーが入力する基本情報は5系統に分解する。service_type(業務形態)、staff(人員配置)、users(利用者構成)、current_addons / near_miss_addons(取得状況)、care_software(利用中システム)、region_class(地域区分=単価に影響)。

1.2 加算マスタスキーマ(AddonMaster)

各加算は「単位数」「算定基準(月1回/回ごと/日ごと)」「必要人員」「LIFE提出要否」「新規雇用が必要な職種とその人件費目安」「初期システム投資目安」を持つレコードとして定義する。v2では以下2つのフィールドを拡張した。

  • requirement_process — 単純な文字列配列から、{task, owner, frequency, deliverable}(内容・担当・頻度・作成物)を持つ構造化レコードの配列に変更。ロードマップ(③)で「誰が・いつ・何を作るか」を明示できるようにした。
  • variable_cost_items(新規) — 既存職員の稼働時間人件費・水道代・消耗品費など、新規雇用の有無に関わらず発生するランニングコストを{label, basis, yen}で保持。basisがper_occurrenceなら算定回数×対象者数に比例、per_monthなら対象者数に関わらず月一定額。

これにより、加算ごとに個別ロジックを書かずに共通アルゴリズムで処理できる。

2

コアアルゴリズム

STEP 0 — 業務形態フィルタ

service_typeがapplicable_servicesに含まれない加算をここで除外する。以降のステップはフィルタ後の候補集合のみに対して実行する。

STEP 1 — ギャップ分析

加算ごとに3種類のギャップを算出する。

  • 人員ギャップ — 必要な常勤換算数と現有人員数の差(不足分)
  • 運用ギャップ — 未整備の計画書作成・アセスメント頻度など手続き数(「ほぼ達成」自己申告があれば軽減)
  • システムギャップ — LIFE提出等が必要な加算で、現行システムのLIFE連携対応度から算出

STEP 2 — 収益試算

年間収益 = 単位数 × 算定頻度(月換算) × 対象利用者数 × 地域区分単価(円/単位) × 12ヶ月。対象利用者数は加算の性質に応じて調整する(例: 認知症専門ケア加算は認知症該当割合のみを対象人数とする)。

STEP 3 — コスト試算(v2で拡張)

v1では「新規雇用が必要な場合の人件費」しか計上しておらず、既存職員が対応できる加算のランニングコストが0円になってしまう問題があった(例: 入浴介助加算(II)で、実際には発生する介助人件費・水道代・消耗品費が抜けていた)。v2では加算ごとにvariable_cost_items(既存職員の稼働時間人件費・水道代・消耗品費など)を費目単位で保持し、初期コスト・新規雇用人件費・研修費・システム対応工数・variable_cost_itemsを費目別の内訳付きで積み上げる。「既に要件を満たしている=追加費用ゼロ」という誤った前提を排除し、既存職員が対応する場合でも稼働時間・水道代・消耗品費は発生するという前提で算出することが、v2でのコスト試算の鍵となる。

STEP 4 — 費用対効果ランキング(①)

年間純利益 = 年間収益 − ランニングコスト(費目別内訳の合計) − (初期コスト÷5年償却)。この値の降順で上位3〜5件を提示し、赤字(純利益がマイナス)の場合は明示的に警告フラグを立てる。

STEP 5 — 難易度評価(④の基礎)

難易度スコア = 人員ギャップ×10 + システムギャップ×3 + 運用ギャップ×1。人員採用は解消に数ヶ月を要し裁量の余地が小さいため最重視し、書類・運用整備は裁量で解消しやすいため軽い重みとする。④のランキングは人員ギャップ=0の候補に限定した上でこのスコアの昇順で並べる。難易度と費用対効果は別軸であり、難易度が低くても純利益がマイナスになる加算がある点は§3で具体例を示す。

STEP 6 — ロードマップ生成(③、v2でフェーズ構造化)

v1は要件を一行ずつ列挙するだけで、誰が・いつ・何を作成するのかが不明確だった。v2では進め方が分かるように5フェーズに再編する。

  • Phase 1 人員体制の整備 — 人員ギャップがある場合のみ。新規雇用/外部委託/既存職員の再配置のいずれかを提示し、想定リードタイムを明示。
  • Phase 2 運用ルールの構築 — 各運用タスクを{task, owner, frequency, deliverable}の構造で保持し、「誰が・どの頻度で・何を作成するか」を明示。
  • Phase 3 システム・記録対応 — LIFE提出が必要な加算のみ。care_softwareのLIFE連携対応度に応じ3段階でアドバイスを出し分ける。
  • Phase 4 届出・算定開始 — 提出書類・提出先・算定開始タイミングの目安を明示。
  • Phase 5 継続運用・見直し — 記録の保存義務(原則5年)、要件を満たさなくなった場合の対応、報酬改定時の見直しを明示。算定して終わりではなく継続運用が必要であることを示す。

入浴介助加算(II)のコスト内訳が示す教訓 v1では「専門職の新規雇用が不要」という理由だけで年間追加コストを0円と算出していた(年間純利益 +5,944,752円)。実際には(1)個浴対応による追加介助人件費 年間2,916,000円、(2)水道・ガス代 年間583,200円、(3)洗浄剤・タオル等消耗品費 年間243,000円、(4)3ヶ月毎の居宅訪問アセスメント人件費 年間360,000円、計年間4,102,200円のランニングコストが発生する。v2ではこれらをvariable_cost_itemsとして明示的に計上し、年間純利益は+1,842,552円に補正された。「既存職員が対応できる=無償」ではないことを示す典型例。
参考:負の費用対効果の例 栄養アセスメント加算を得るために管理栄養士1名を新規雇用した場合、年間収益は約30万円+アセスメント用品費に対し、人件費は約450万円かかるため、年間純利益は約 −4,210,560円となり赤字判定となる。他事業所からの巡回・外部委託等で人員要件を満たせないか、先に検討すべき典型例。

② コスト見積もり(内訳付き・抜粋)

初期コストとランニングコストを、費目(新規雇用人件費・研修費・LIFE対応工数・既存職員の稼働時間人件費・水道代・消耗品費など)ごとの内訳付きで候補全件に提示する。「人員ギャップがゼロ=追加コストもゼロ」という誤った前提を排除し、既存職員が対応する加算にも会議・記録・巡回等の稼働時間コストが計上される。

③ 獲得ロードマップ(個別機能訓練加算(I)ロの例、Phase構造)

Phase 1: 人員体制の整備(想定リードタイム: 常勤・非常勤いずれも新規採用が必要。求人掲載から入職まで平均2〜3ヶ月を想定)

  • 機能訓練指導員(PT/OT/ST等)を常勤換算 1.00 名相当、新規雇用・外部委託・既存職員の再配置のいずれかで確保する。

Phase 2: 運用ルールの構築

  • 個別機能訓練計画書の作成 — 担当: 機能訓練指導員 / 頻度: 利用開始時 / 作成物: 個別機能訓練計画書
  • 目標達成度の評価・計画の見直し — 担当: 機能訓練指導員 / 頻度: 3ヶ月ごと / 作成物: 評価記録・改定後の計画書
  • 生活機能向上を目的とした訓練の実施記録 — 担当: 機能訓練指導員/介護職員 / 頻度: 訓練実施の都度 / 作成物: 訓練実施記録

Phase 4: 届出・算定開始

  • 体制等状況一覧表(様式は管轄自治体のホームページで公開)に必要事項を記入し、加算根拠となる計画書等の写しを添えて提出する。
  • 提出先は事業所所在地を管轄する都道府県または市町村の介護保険担当課。
  • 算定開始日は提出月の翌月1日からとする運用が一般的だが、詳細は自治体の運用に従うこと。

Phase 5: 継続運用・見直し

  • Phase 2で整備した記録を継続的に作成・保管する(指定基準上、原則5年間の保存義務)。
  • 要件を満たさなくなった場合は速やかに算定を中止し、必要に応じて変更届を提出する。
  • 報酬改定(3年ごと)・処遇改善加算等の毎年の運用見直しのタイミングで、本マスタの単位数・要件を再確認する。

Phase 3(システム対応)は本加算ではLIFE提出が不要のため生成されない。要件がない場合はフェーズ自体を省略する設計。

④ 難易度が低い加算ランキング

人員ギャップがゼロの候補に限定し、難易度スコア昇順で並べる。

加算名難易度スコア年間純利益
1口腔・栄養スクリーニング加算(I)1.0+109,296円
2科学的介護推進体制加算(I)2.6+126,992円

    support1: int = 0
    support2: int = 0
    care1: int = 0
    care2: int = 0
    care3: int = 0
    care4: int = 0
    care5: int = 0
    dementia_ratio: float = 0.0

    @property
    def total_users(self) -> int:
        return (self.support1 + self.support2 + self.care1 + self.care2
                + self.care3 + self.care4 + self.care5)

    @property
    def total_care_only(self) -> int:
        return self.care1 + self.care2 + self.care3 + self.care4 + self.care5

    @property
    def mid_to_high_care_ratio(self) -> float:
        denom = self.total_care_only
        if denom == 0:

        return (self.care3 + self.care4 + self.care5) / denom

# 2. コアアルゴリズム
# ---------------------------------------------------------------------------

def filter_by_service(facility: FacilityInput) -> list:
    """STEP0: 業務形態に存在しない加算を除外する。"""
    return [a for a in ADDON_MASTER if facility.service_type in a.applicable_services]


def compute_requirement_gap(facility: FacilityInput, addon: AddonMaster) -> dict:
    """STEP1: 現状の人員・利用者構成と加算要件の差分(ギャップ)を計算する。"""
    staff = facility.staff
    gap = {"staffing_gap": 0.0, "process_gap": 0, "system_gap": 0.0}

    for role, required in addon.requirement_staff.items():
        current = getattr(staff, role, 0)
        if current < required:
            gap["staffing_gap"] += (required - current)

    gap["process_gap"] = len(addon.requirement_process)

    if addon.requires_life_submission:
        life_support = CARE_SOFTWARE_LIFE_SUPPORT.get(facility.care_software, 0.3)
        gap["system_gap"] = round((1 - life_support), 2)

    return gap


def estimate_annual_revenue(facility: FacilityInput, addon: AddonMaster) -> float:
    """STEP2: 加算による想定年間請求増加額を算出する。"""
    total_users = facility.users.total_users or 1

    if addon.code == "NINCHISHOU_SENMON_I":
        eligible = total_users * facility.users.dementia_ratio
    elif addon.code == "NICHIJO_KEIZOKU":
        eligible = facility.users.care4 + facility.users.care5
    else:
        eligible = total_users

    monthly_units = addon.unit_value * addon.monthly_frequency * eligible
    monthly_yen = monthly_units * facility.unit_price
    return round(monthly_yen * 12, 0)


def estimate_annual_cost(facility: FacilityInput, addon: AddonMaster, gap: dict) -> dict:
    """STEP3: 初期コストとランニングコストを見積もる。"""
    running_cost = 0.0
    initial_cost = addon.system_initial_cost

    if gap["staffing_gap"] > 0 and addon.hiring_needed_role:
        running_cost += addon.hiring_annual_cost * min(gap["staffing_gap"], 1.0)

    running_cost += addon.training_annual_cost

    if gap["system_gap"] > 0:
        running_cost += gap["system_gap"] * 100000

    return {"initial_cost": initial_cost, "running_cost_annual": running_cost}


def compute_difficulty_score(gap: dict) -> float:
    """STEP5: 難易度スコア。値が小さいほど「獲得しやすい」。"""
    return (gap["staffing_gap"] * 10.0
            + gap["system_gap"] * 3.0
            + gap["process_gap"] * 1.0)


def generate_roadmap(facility: FacilityInput, addon: AddonMaster, gap: dict) -> list:
    """STEP6: 要件クリアへのロードマップを順序立てて生成する。"""
    steps = []
    step_no = 1

    if gap["staffing_gap"] > 0:
        steps.append(f"{step_no}. 【人員】{addon.hiring_needed_role or '必要職種'}を"
                      f"常勤換算 {gap['staffing_gap']:.2f} 名相当、新規雇用または外部連携で確保する。")
        step_no += 1

    for process in addon.requirement_process:
        steps.append(f"{step_no}. 【運用】{process}")
        step_no += 1

    if addon.requires_life_submission:
        life_support = CARE_SOFTWARE_LIFE_SUPPORT.get(facility.care_software, 0.3)
        if life_support >= 0.8:
            sys_note = f"{facility.care_software}はLIFE連携機能を標準搭載。設定を有効化するだけで対応可能。"
        elif life_support >= 0.5:
            sys_note = f"{facility.care_software}はエクスポート対応済みだが手動マッピングが必要。"
        else:
            sys_note = f"{facility.care_software}は連携が弱いため専用ツール導入を検討。"
        steps.append(f"{step_no}. 【システム】{sys_note}")
        step_no += 1

    steps.append(f"{step_no}. 【提出】都道府県・市町村へ体制等状況一覧表を提出する。")
    return steps


@dataclass
class AddonResult:
    addon: AddonMaster
    gap: dict
    annual_revenue: float
    initial_cost: float
    running_cost_annual: float
    net_annual_profit: float
    difficulty_score: float
    roadmap: list


def analyze(facility: FacilityInput) -> list:
    candidates = filter_by_service(facility)
    results = []

    for addon in candidates:
        if addon.code in facility.current_addons:
            continue

        gap = compute_requirement_gap(facility, addon)
        if addon.code in facility.near_miss_addons:
            gap["process_gap"] = max(0, gap["process_gap"] - 1)

        revenue = estimate_annual_revenue(facility, addon)
        cost = estimate_annual_cost(facility, addon, gap)
        amortized_initial = cost["initial_cost"] / 5
        net_profit = revenue - cost["running_cost_annual"] - amortized_initial
        difficulty = compute_difficulty_score(gap)
        roadmap = generate_roadmap(facility, addon, gap)

        results.append(AddonResult(
            addon=addon, gap=gap, annual_revenue=revenue,
            initial_cost=cost["initial_cost"],
            running_cost_annual=cost["running_cost_annual"],
            net_annual_profit=net_profit, difficulty_score=difficulty,
            roadmap=roadmap,
        ))

    return results


def build_report(facility: FacilityInput, top_n: int = 5) -> dict:
    """4つの出力(①②③④)をまとめて生成する。"""
    results = analyze(facility)

    cost_effectiveness_rank = sorted(
        results, key=lambda r: r.net_annual_profit, reverse=True)[:top_n]

    easy_candidates = [r for r in results if r.gap["staffing_gap"] == 0]
    ease_rank = sorted(easy_candidates, key=lambda r: r.difficulty_score)[:top_n]

    return {
        "① 費用対効果ランキング": [
            {"加算名": r.addon.name, "年間増収見込み": f"{r.annual_revenue:,.0f}円",
             "年間追加コスト": f"{(r.running_cost_annual + r.initial_cost/5):,.0f}円",
             "年間純利益": f"{r.net_annual_profit:,.0f}円",
             "赤字警告": r.net_annual_profit < 0}
            for r in cost_effectiveness_rank
        ],
        "② コスト見積もり": [
            {"加算名": r.addon.name, "初期コスト": f"{r.initial_cost:,.0f}円",
             "ランニングコスト(年間)": f"{r.running_cost_annual:,.0f}円"}
            for r in results
        ],
        "③ 獲得ロードマップ": [
            {"加算名": r.addon.name, "手順": r.roadmap} for r in cost_effectiveness_rank
        ],
        "④ 難易度が低い加算ランキング": [
            {"加算名": r.addon.name, "難易度スコア": round(r.difficulty_score, 1),
             "年間純利益": f"{r.net_annual_profit:,.0f}円"}
            for r in ease_rank
        ],
    }

5

運用上の注意

  • 単位数の更新体制 — マスタは3年に一度の報酬改定に加え、処遇改善加算のように毎年運用が変わるものもあるため、更新責任者と更新タイミングを社内規程として明文化する。
  • 地域区分・サービス種別ごとの単価差 — 本実装は簡略化した地域区分単価を採用しているが、実際は1級地〜その他の8区分に加えサービス種別ごとに単価係数が異なる。本番マスタでは区分×サービス種別のテーブルを持たせること。
  • 併算定制限・逓減制 — 一部の加算は同時算定不可(併給調整)や利用者数に応じた逓減制が存在する。STEP2の収益試算にはこれらの制約を追加考慮するロジックを組み込む必要がある。
  • 対象利用者数の精緻化 — 本サンプルでは加算ごとに簡略化した対象者算出ルールを用いている。実運用では加算ごとに対象者算出関数を個別定義するのが望ましい。
  • 既存業務システムとの連携 — カイポケ・ワイズマン・ほのぼの・ケアカルテ等のCSVエクスポート機能の対応範囲は頻繁にアップデートされるため、CARE_SOFTWARE_LIFE_SUPPORTの値は各社の最新仕様書で定期検証する。

本設計書はロジック・アルゴリズムの提示を目的とした参考資料であり、個別の加算算定可否を保証するものではない。実際の算定にあたっては、指定権者(都道府県・市町村)への確認および専門家(社会保険労務士・行政書士等)への相談を推奨する。
